# 가계부(예산 계산기) 앱 — 설계 문서

작성일: 2026-07-12

## 1. 목적과 배경

매달 지출은 **고정비**(자동이체·학원·보험·용돈·대출·저금, 그리고 카드로 빠지는 고정 결제)와
**고정비 외 카드 지출**(현금 부족·여행 등으로 현대/신한/삼성카드를 쓰게 되는 결제)로 나뉜다.

사용자는 이미 엑셀로 월 예산을 계산하고 있으며, 이 앱은 그 엑셀의 계산을 **폰에서 쓰는 현대적 UI(뱅크샐러드풍)** 로 옮기는 것이 목표다.

**1단계(이 문서 범위):** 고정비를 직접 입력해두고, 이번달 실제 카드값만 넣으면
`총 필요 예산 / 잔여금액 / 고정비 외 카드지출`을 자동으로 도출하는 예산 계산기.

**Phase 2(미래, 이 문서 범위 밖):** 안드로이드 카드 승인 문자(SMS)를 자동 수집해
"이번달 실제 카드값"을 자동으로 채우고, 고정비 외 결제를 자동 분류. 같은 데이터 구조/DB를 그대로 확장.

## 2. 핵심 계산 모델

고정비를 **결제수단**으로 두 갈래로 나누는 것이 핵심이다.

- **현금이체 고정비**(현금이체 / 현금이체(자동)) → 매달 동일. 그대로 합산 = `현금이체 합계`
- **카드 고정비**(현대 / 신한 / 삼성) → 카드로 나가야 할 "고정비 기준선"

```
현금이체 합계        = Σ(현금이체 · 현금이체(자동) 고정비)
카드 고정비 기준선(카드별) = Σ(해당 카드 고정비)
총 필요 예산         = 현금이체 합계 + Σ(이번달 실제 카드값)   ← 실제 카드값은 사용자가 입력
잔여금액            = 수입합계 − 총 필요 예산
고정비 외 카드지출(카드별) = 이번달 실제 카드값 − 카드 고정비 기준선
```

이 모델은 엑셀의 `총 필요 예산 / 잔여금액` 계산을 그대로 재현하면서,
동시에 사용자의 원래 관심사인 **"고정비 외에 카드로 얼마나 더 썼나"** 를 자동으로 산출한다.

### 3.4 월 히스토리
`monthly_card_actuals`에 월별 실제 카드값이 쌓이고, 고정비/수입 스냅샷과 결합해 월별 추이를 만든다.
(1단계는 현재 고정비/수입 기준으로 계산. 과거 고정비 변경 이력 추적은 범위 밖.)

## 4. 화면 구성 (뱅크샐러드풍, 모바일 세로 우선)

- **화면 1 — 이번달 대시보드(메인)**: 큰 숫자 카드(총 필요 예산 / 잔여금액, +/− 색상 구분),
  카드별 요약(고정비 기준선 / 실제 입력값 / 고정비 외 지출), 카테고리 도넛 차트,
  "이번달 실제 카드값 입력" (현대·신한·삼성).
- **화면 2 — 고정비 관리**: 결제수단·카테고리별 접이식 리스트, 추가/수정/삭제,
  '변동' 배지, 하단에 현금이체 합계 / 카드 고정비 기준선 실시간 표시.
- **화면 3 — 수입 관리**: 수입 항목 추가/수정, 수입합계.
- **화면 4 — 월 히스토리**: 지난달들의 총 필요 예산·잔여금액·고정비 외 지출 추이.

디자인 톤: 흰 배경 + 부드러운 카드 그림자 + 파스텔 포인트 컬러 + 큰 숫자 타이포.

## 5. 기술 스택 & 배포

- **프론트엔드**: React + Vite + TypeScript + Tailwind CSS + Recharts. **PWA**(폰 홈 화면 설치).
- **데이터**: **Supabase**(무료 클라우드 Postgres). PC와 폰이 같은 DB에 접속 → **자동 연동**.
- **배포**: **GitHub Pages**(정적 호스팅). 사용자가 이미 GitHub 사용 중.
- **접근 제한**: 개인 가계부이므로 Supabase RLS + 간단한 접근키/패스코드로 보호.
- **백업**: 데이터 내보내기/불러오기(JSON) 기능 기본 탑재.

### 5.1 왜 이 구성인가
- 폰 우선 + PC 연동 + 무료라는 세 조건을 동시에 만족.
- GitHub Pages는 정적 파일만 호스팅 가능 → 백엔드 대신 Supabase 클라이언트로 DB 직접 접속.
- Phase 2(문자 자동수집) 확장 시 같은 Supabase DB를 그대로 사용.

### 6.3 명세서 대조 시 발견된 참고사항
- **디즈니 = 카드표기 "페이플"**, **스카이라이프 = "KT스카이라이프"**, **구글원 = "Google Digital Inc."** 등 카드표기명과 사용자 이름이 다름 → Phase 2 자동매칭 때 별칭 사전 필요.
- 6·7월 명세서상 **현대해상_정한** 실제 청구는 31,940 / 10,320(엑셀은 7,620)로 일부 상이. 시드는 엑셀값을 따르고 사용자가 확정.
- 6·7월 **현대카드**는 여행 지출(소노인터내셔널·코스트코·주유·하이패스 등)이 많아 "고정비 외 카드지출"이 크게 발생 → 이 앱이 풀려는 문제가 실제로 존재함을 데이터로 확인.

## 7. 범위 밖 (YAGNI)
- 카드사 자동 로그인 스크래핑 (취약·약관위반) → 채택 안 함.
- SMS 자동 수집/자동 분류 → Phase 2.
- 생활비 체크카드 상세 내역 관리 → 관리 대상 아님.
- 다중 사용자/권한, 과거 고정비 변경 이력 추적 → 1단계 범위 밖.
